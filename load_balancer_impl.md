# ロードバランサーの実装について

このシミュレーターでは、実際のインフラを使わず**数式による計算**でロードバランサーの挙動を再現しています。

---

## 実装の核心：負荷の均等分散を数式で表現

実際のロードバランサーは「届いたリクエストを複数のサーバーに振り分ける」装置ですが、このシミュレーターでは以下の1行でその効果を表現しています（`simulation.py:79`）。

```python
cpu_per_instance = min(users / (current_instances * 30), 100.0)
```

**考え方：**
- サーバーが1台のとき → 全アクセスが1台に集中
- サーバーが3台のとき → 全アクセスが3台に均等分散

| 同時アクセス数 | インスタンス数 | 1台あたりの負荷 |
|---|---|---|
| 3,000人 | 1台 | 100%（限界） |
| 3,000人 | 3台 | 33%（余裕あり） |
| 12,000人 | 6台 | 67%（安定） |

これはラウンドロビン方式（順番に均等割り当て）のロードバランサーを理想的に単純化したモデルです。

---

## オートスケーリングの実装

ロードバランサー単体では「振り分け先を増やす」機能は持ちません。実際のAWS / Azureでは、負荷に応じてサーバー台数を自動で増減する**オートスケーリング**がセットで動作します。このシミュレーターでも同様の仕組みを実装しています（`simulation.py:82〜91`）。

```python
if (
    cpu_per_instance > AWS_SCALE_THRESHOLD_CPU   # CPU使用率が70%を超えたら
    and new_instances < AWS_MAX_INSTANCES         # 最大台数（8台）未満なら
    and new_cooldown == 0                         # クールダウン中でなければ
):
    new_instances = min(current_instances + 1, AWS_MAX_INSTANCES)
    new_cooldown = AWS_SCALE_DELAY_STEPS          # 次のスケールまで3ステップ待つ
    cpu_per_instance = min(users / (new_instances * 30), 100.0)  # 負荷を再計算
```

**3つの判定条件：**

| 条件 | 意味 | 実際のAWSでの対応 |
|---|---|---|
| CPU > 70% | 各サーバーが高負荷 | CloudWatchのCPUメトリクス監視 |
| 台数 < 8台 | 上限に達していない | Auto Scalingグループの`maxSize` |
| クールダウン = 0 | 直前のスケールが落ち着いた | スケーリングポリシーのクールダウン期間 |

---

## 応答時間への反映

ロードバランサーによって負荷が分散されると、応答時間も改善されます（`simulation.py:94`）。

```python
response_time = 0.2 + (users / (new_instances * 3000)) * 0.8
```

- `0.2` は最低応答時間（ベースレイテンシ）
- `users / (new_instances * 3000)` の部分がインスタンス数に反比例 → 台数が増えるほど応答時間が下がる

---

## Streamlit Cloud との比較

Streamlit Cloudでは `current_instances` が常に1固定のため、アクセスが増えるほど1台に負荷が集中し、応答時間が指数的に悪化してクラッシュします（`simulation.py:37`）。

```python
# Streamlit Cloud: 1台固定、分散なし
response_time = 0.2 + (users / 1000) ** 2 * 0.2  # 指数的に悪化
```

```python
# AWS / Azure: インスタンス数で割ることで負荷分散を表現
response_time = 0.2 + (users / (new_instances * 3000)) * 0.8  # 台数増加で安定
```

---

## まとめ

| 要素 | 実装方法 |
|---|---|
| ロードバランサーの分散効果 | `users ÷ インスタンス数` で1台あたりの負荷を計算 |
| オートスケーリング | CPU使用率が閾値超えで台数を+1（クールダウンあり） |
| 応答時間の安定化 | 台数に反比例する応答時間の計算式 |

実際のインフラ構築は不要で、**「分散 = 割り算」** という単純な数理モデルで直感的に再現しています。
